# Development Workflow: как держать голову чистой

Главный принцип всего: **один контекст за раз**. Пока ты в бэкенде — фронтенд не существует. Пока пишешь тест — не думаешь о UI.

---

## Мета-workflow: как брать фичу в работу

Каждая фича проходит через одну и ту же последовательность. Записывай её как чеклист в задаче:

```
[ ] 1. Доменная модель (если нужна)
[ ] 2. Репозиторий
[ ] 3. Сервис
[ ] 4. API эндпоинт
[ ] 5. Unit-тест на домен/сервис
[ ] 6. Integration-тест на эндпоинт
[ ] 7. API hook на фронте
[ ] 8. UI компонент
[ ] 9. Страница/feature
[ ] 10. Ручная проверка в браузере
```

Ты всегда знаешь, где ты находишься. Нет ощущения "что там ещё осталось".

---

## Workflow для Backend-фичи

### Шаг 1 — Domain (если нужен новый объект или алгоритм)

Открываешь только `app/domain/`. Больше ничего.

Вопросы, которые держишь в голове:

- Что это за сущность? Какие у неё данные?
- Какое поведение (методы) у неё есть?
- Какие инварианты она должна соблюдать?

Пишешь `@dataclass` или обычный класс. Никаких импортов из `models`, `services`, `api`. Если поймал себя на том, что тянешь зависимость вверх — это сигнал, что логика не туда.

---

### Шаг 2 — Repository

Открываешь только `app/repositories/`. Задаёшь себе один вопрос:

> "Какие SQL-операции нужны сервису?"

Пишешь интерфейс метода, потом реализацию. Никакой логики. Только `get`, `save`, `delete`, `list`. Если начинаешь писать `if` внутри репозитория — это логика, которая должна быть в сервисе.

---

### Шаг 3 — Service

Открываешь только `app/services/`. Сервис — это оркестратор:

```
получить данные → вызвать доменный метод → сохранить результат
```

Если метод сервиса вырос больше 30 строк — скорее всего часть логики просится в домен.

---

### Шаг 4 — Schema + Router

Сначала схемы (`app/schemas/`): что принимает эндпоинт, что возвращает. Только потом роутер. Роутер — это тонкая обёртка: распаковать запрос → вызвать сервис → вернуть ответ. Никакой логики в роутере.

---

### Шаг 5 — Тесты

Два вида тестов, разный контекст:

**Unit-тесты** (`tests/unit/`) — только домен и сервис с замоканными репозиториями. Запускаются мгновенно, без БД. Фокус: "правильно ли работает алгоритм?"

```python
# Паттерн для unit-теста сервиса
def test_plan_excludes_mastered():
    mock_repo = Mock()
    mock_repo.get_mastered_ids.return_value = {skill_b.id}
    service = PlanService(skill_repo=mock_repo, ...)
    ...
```

**Integration-тесты** (`tests/integration/`) — реальная БД (тестовая), реальный HTTP-клиент через `httpx.AsyncClient`. Фокус: "работает ли эндпоинт от и до?"

```python
# Паттерн для интеграционного теста
async def test_create_plan_returns_201(client, seeded_skills):
    response = await client.post("/api/v1/plans", json={...})
    assert response.status_code == 201
    assert "id" in response.json()
```

Правило: **unit-тесты пишешь сразу после домена/сервиса**, пока контекст ещё горячий. Integration-тесты — после роутера.

## Workflow для Frontend-фичи

Фронтенд начинается только после того, как эндпоинт работает и задокументирован в Swagger. Это важно — не параллельно, а после. Иначе будешь одновременно отлаживать два слоя.

### Шаг 1 — Типы

Открываешь `src/shared/types/index.ts`. Смотришь в Swagger на схему ответа. Пишешь/дополняешь TypeScript-интерфейс. Всё.

---

### Шаг 2 — API Hook

Открываешь `src/shared/api/hooks.ts`. Пишешь `useQuery` или `useMutation` для нового эндпоинта. Проверяешь в DevTools, что данные приходят. На этом этапе UI не существует — ты проверяешь только сеть.

---

### Шаг 3 — Компонент без стилей

Пишешь компонент, который просто рендерит данные. Никакого Tailwind, никакого дизайна. Только `<div>{data.title}</div>`. Цель — убедиться, что логика и состояние работают.

---

### Шаг 4 — Стили и UX

Только теперь открываешь Tailwind и думаешь о внешнем виде. Голова занята только визуальным — данные уже работают.

---

### Шаг 5 — Страница

Собираешь компоненты в страницу, подключаешь роутинг. Ничего нового не пишешь — только компонуешь готовое.

---

## Workflow для работы с графом (специфика проекта)

Граф — самая сложная часть. Для неё отдельный режим:

**Когда меняешь алгоритм** — работаешь только в `domain/skill_graph.py` и `tests/test_skill_graph.py`. Открываешь два файла рядом, больше ничего. Запускаешь `pytest tests/test_skill_graph.py -v` после каждого изменения.

**Когда меняешь визуализацию** — работаешь только в `features/graph/`. Backend не трогаешь. Если нужны другие данные — сначала закрываешь задачу по визуализации, потом открываешь задачу по бэкенду.

---

## Workflow для отладки бага

Баг — это отдельный контекст, не смешивается с разработкой фичи.

```
1. Воспроизвести баг минимальным сценарием
2. Определить слой: Domain / Service / Repository / API / Frontend
3. Написать падающий тест, который воспроизводит баг
4. Починить
5. Убедиться, что тест зелёный
6. Запустить все тесты
```

Правило: **сначала тест, потом фикс**. Это единственный способ быть уверенным, что баг не вернётся.

---

## Правила переключения контекста

Это самое важное для "чистой головы":

**Перед тем как переключиться** — зафиксируй текущий контекст. Буквально напиши в заметки: "остановился здесь, следующий шаг — X". Это 30 секунд, которые экономят 10 минут на восстановление.

**Никогда не держи одновременно открытыми** файлы бэкенда и фронтенда. Один редактор — один слой.

**Коммит как граница контекста.** Сделал шаг из чеклиста — сделал коммит с понятным сообщением. Коммит закрывает контекст. Следующий коммит — новый контекст.

```bash
git commit -m "feat(domain): add get_subgraph with mode support"
git commit -m "feat(service): plan builder uses subgraph + topo sort"
git commit -m "feat(api): POST /plans endpoint"
git commit -m "test(unit): skill_graph subgraph and cycle detection"
```

Глядя на историю коммитов, ты всегда видишь, на каком шаге чеклиста ты находишься.

## Шпаргалка: какой файл открывать при каком вопросе

| Вопрос                          | Файл                                    |
| ------------------------------- | --------------------------------------- |
| Правильно ли работает алгоритм? | `domain/skill_graph.py` + тест          |
| Нужен новый SQL-запрос?         | `repositories/`                         |
| Нужна бизнес-логика?            | `services/`                             |
| Нужен новый эндпоинт?           | `api/v1/` + `schemas/`                  |
| Нужны новые данные на фронте?   | `shared/types/` → `shared/api/hooks.ts` |
| Нужен новый UI-блок?            | `features/<feature>/`                   |
| Нужна новая страница?           | `pages/`                                |
| Баг в данных?                   | Начинай с домена, спускайся вниз        |
| Баг в отображении?              | Начинай с хука, поднимайся вверх        |
